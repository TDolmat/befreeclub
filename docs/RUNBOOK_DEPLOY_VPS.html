Deploy na nowy VPS

Krok po kroku: stawiasz nowy serwer od zera, hartujesz go, wrzucasz panel admina (FastAPI + nginx + Postgres za Caddy) i przenosisz dane ze starego boxa. Landing robimy osobno w fazie 2. Komendy są do kopiowania. Checkboxy zapamiętują postęp w przeglądarce.

Legenda hostów
Przy każdym bloku komend jest etykieta gdzie to odpalić: NA MACU Twój laptop   NOWY VPS świeży serwer   STARY VPS obecny box z adminem   DNS panel Hostinger.
Podmieniaj WIELKIE_SŁOWA (np. NOWE_IP, DB_PASS) na swoje wartości.

iPrzegląd

W tej fazie przenosimy tylko panel admina (admin.befreeclub.pro). Stary VPS z adminem, befreeclub-api i gigscope zostaje włączony przez całą operację - to Twój rollback. Wygaszasz go dopiero, gdy nowy panel pochodzi kilka dni bez problemów.

iiZanim zaczniesz - co miej pod ręką

  • IP nowego VPS + hasło root (albo klucz SSH ustawiony przy zamówieniu OVH). To NOWE_IP.
  • Dostęp SSH do starego VPS (do dumpa danych). To STARE_IP + Twój user.
  • Twój klucz SSH na Macu (~/.ssh/id_ed25519.pub). Jak nie masz: ssh-keygen -t ed25519.
  • Dostęp do DNS w Hostinger (domena befreeclub.pro, rekord A dla admin).
  • Circle Headless token (ten sam co w starym befreeclub-api / starym adminie, zmienna BOOTSTRAP_ADMIN_TOKEN). Bez niego Circle DM nie zaloguje się do Circle.
  • Login do Claude Max (zalogujesz CLI w kontenerze przez device flow).
  • Hasło DB starego boxa (DB_PASS w ~/repos/befreeclub/admin/.env na starym VPS) - do dumpa.
Zrób dzień wcześniej
Obniż TTL rekordu A admin.befreeclub.pro w Hostinger do 300 s (5 min). Wtedy przepięcie DNS w kroku 10 zadziała w minuty, nie w godziny.

2User bez roota + klucz SSH

Nie pracujemy na roocie. Tworzymy usera tomek z sudo i wgrywamy do niego Twój klucz.

NOWY VPS
adduser tomek            # ustaw hasło, resztę pól Enter

NA MACU
# Skopiuj swój klucz publiczny na nowego usera
ssh-copy-id tomek@NOWE_IP

Wskazówka
Od teraz loguj się tylko jako tomek. Roota użyjesz przez sudo.

3Hardening SSH + firewall

Trzy rzeczy: wyłączamy logowanie roota i hasłem (tylko klucz), zapora przepuszcza tylko SSH/HTTP/HTTPS, fail2ban banuje próby zgadywania.

NOWY VPS
# Firewall - najpierw przepuść SSH, potem włącz (inaczej się zablokujesz)
sudo ufw allow OpenSSH
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw enable
sudo ufw status
# Hardening SSH: dopisz na końcu pliku konfiguracji
sudo tee /etc/ssh/sshd_config.d/10-bfc.conf > /dev/null <<'EOF'
PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
EOF
sudo systemctl restart ssh
Zanim zamkniesz terminal
Otwórz drugą sesję ssh tomek@NOWE_IP i sprawdź, że wchodzi. Jak działa - dopiero wtedy zamykaj pierwszą. Inaczej przy błędzie w configu zostajesz na zewnątrz. 
# fail2ban - domyślny jail SSH wystarcza, włącz i zostaw
sudo systemctl enable --now fail2ban
# Automatyczne aktualizacje bezpieczeństwa
sudo apt -y install unattended-upgrades
sudo dpkg-reconfigure -plow unattended-upgrades

5Caddy na hoście (reverse proxy + SSL)

Caddy stoi osobno i sam ogarnia certyfikaty Let's Encrypt. Na razie stawiamy go z pustą konfiguracją - blok domeny dodamy dopiero po wgraniu aplikacji (krok 10), bo cert wymaga działającego backendu i DNS celującego w nowy box.

NOWY VPS
mkdir -p ~/caddy && cd ~/caddy

Utwórz ~/caddy/docker-compose.yml:

name: caddy
services:
  caddy:
    image: caddy:2-alpine
    container_name: caddy
    restart: unless-stopped
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./Caddyfile:/etc/caddy/Caddyfile:ro
      - caddy-data:/data
      - caddy-config:/config
    networks:
      - caddy_network
volumes:
  caddy-data:
  caddy-config:
networks:
  caddy_network:
    external: true
# Pusty Caddyfile na start (blok admina dodasz w kroku 10)
echo "# bloki domen dodawane w kroku 10" > ~/caddy/Caddyfile
docker compose up -d
docker compose ps
Czemu pusto
Gdyby Caddy dostał teraz blok admin.befreeclub.pro, próbowałby wyrobić certyfikat, a DNS jeszcze wskazuje stary box. Najpierw aplikacja, potem DNS, na końcu blok Caddy.

6Kod na serwer

Monorepo trafia na serwer przez prywatne repo GitHub + deploy key (czysto i powtarzalnie przy aktualizacjach). Najpierw jednak musisz zrobić z katalogu repo i wypchnąć go na GitHub.

Najpierw .gitignore - krytyczne
Repo zawiera pliki .env z sekretami i .venv / node_modules. Nie mogą trafić do gita. Upewnij się, że root .gitignore wyklucza: .env, *.env (poza *.example), .venv/, node_modules/, data/, .dev-outbox/, __pycache__/.
NA MACU
cd ~/repos/befreeclub/migracja/befreeclub
# Init repo i pierwszy commit (zrób to Ty - commitujemy jako Ty)
git init
git add -A
git status            # SPRAWDŹ: żadnego .env na liście!
git commit -m "init befreeclub monorepo"

Załóż prywatne repo na GitHub (np. befreeclub) i wypchnij. Potem na serwerze tworzysz deploy key (klucz tylko do tego repo, read-only):

NOWY VPS
ssh-keygen -t ed25519 -f ~/.ssh/befreeclub_deploy -N ""
cat ~/.ssh/befreeclub_deploy.pub
# ^ wklej w GitHub: repo > Settings > Deploy keys > Add (read-only)
cat >> ~/.ssh/config <<'EOF'
Host github-bfc
  HostName github.com
  User git
  IdentityFile ~/.ssh/befreeclub_deploy
  IdentitiesOnly yes
EOF
mkdir -p ~/repos/befreeclub && cd ~/repos/befreeclub
git clone git@github-bfc:TWOJ_LOGIN/befreeclub.git befreeclub
cd befreeclub
Bez GitHuba (alternatywa)
Jak nie chcesz repo na GitHub, wgraj kod bezpośrednio: rsync -av --exclude .venv --exclude node_modules --exclude '*.env' ~/repos/befreeclub/migracja/befreeclub/ tomek@NOWE_IP:~/repos/befreeclub/befreeclub/ z Maca. Aktualizacje robisz wtedy ponownym rsync zamiast git pull.

7Sekrety (.env)

Jeden plik sekretów: backend/.env. Compose czyta go i przez symlink infra/.env bierze DB_PASS do interpolacji.

NOWY VPS
cd ~/repos/befreeclub/befreeclub
cp backend/.env.production.example backend/.env
# Wygeneruj mocne hasło Postgresa
openssl rand -hex 24        # skopiuj wynik
nano backend/.env

W backend/.env ustaw na fazę 1 tylko to:

ZmiennaWartość
DB_PASSwynik openssl rand -hex 24
BOOTSTRAP_ADMIN_TOKENTwój Circle Headless token. Wymagany, żeby Circle DM mintował JWT i gadał z Circle.
OPENAI_API_KEYopcjonalnie - włącza transkrypcje głosówek i opisy zdjęć. Puste = te funkcje wyłączone, reszta działa.

Wszystkie zmienne fazy 2 (STRIPE_*, RESEND_*, CIRCLE_*, SENDER_*, META_*) zostaw puste - landing migrujemy później, a backend startuje z nimi pustymi bez błędu.

# Symlink, żeby compose widział DB_PASS przy interpolacji
cd infra
ln -s ../backend/.env .env
# Zamknij plik sekretów przed światem
chmod 600 ../backend/.env
Reguła sekretów
backend/.env nigdy nie wchodzi do gita ani nigdzie poza ten serwer. chmod 600 = czyta go tylko Twój user. Sprawdź: ls -l backend/.env ma pokazać -rw-------.

9Migracja danych ze starego boxa

To moment z oknem serwisowym. Zatrzymujemy starego admina (żeby dane były ostateczne), robimy dump, przenosimy na nowy box, wgrywamy do bazy tymczasowej i odpalamy skrypt mapujący do nowych schematów.

9a. Stop starego admina + dump

STARY VPS
ssh STARE_IP
cd ~/repos/befreeclub/admin
docker compose stop admin-server      # start okna serwisowego - admin off
mkdir -p ~/backups
docker compose exec -T admin-db pg_dump -U admin -d bfc_admin -Fc \
  > ~/backups/bfc_admin_$(date +%F).dump
ls -lh ~/backups

9b. Przenieś dump na nowy box

NA MACU
# Pobierz dump ze starego i wyślij na nowy (przez Twój laptop)
scp STARE_IP:~/backups/bfc_admin_*.dump /tmp/
scp /tmp/bfc_admin_*.dump tomek@NOWE_IP:~/

9c. Wgraj dump do bazy tymczasowej na nowym boxie

NOWY VPS
cd ~/repos/befreeclub/befreeclub/infra
# Baza tymczasowa na surowe dane ze starego schematu (public.*)
docker compose exec -T db createdb -U befreeclub bfc_admin_src
# Wgraj dump (--no-owner bo user nazywa się inaczej niż na starym boxie)
cat ~/bfc_admin_*.dump | docker compose exec -T db \
  pg_restore -U befreeclub --no-owner --no-acl -d bfc_admin_src

9d. Uruchom skrypt migracji

Skrypt mapuje stare tabele (public.*) na nowe schematy (admin.*, circle_dm.*) i przenosi je do bazy befreeclub. Odpalamy go na hoście, oba Postgresy widzi po loopbacku 127.0.0.1:5434.

NOWY VPS
# uv do odpalenia skryptu (jednorazowo)
curl -LsSf https://astral.sh/uv/install.sh | sh
source ~/.bashrc
cd ~/repos/befreeclub/befreeclub/scripts/data-migration
# DRY-RUN: pokazuje plan i liczniki, nic nie zapisuje. DB_PASS = to z backend/.env
uv run --with asyncpg python migrate_data.py \
  --source-dsn "postgresql://befreeclub:DB_PASS@127.0.0.1:5434/bfc_admin_src" \
  --target-dsn "postgresql://befreeclub:DB_PASS@127.0.0.1:5434/befreeclub" \
  --dry-run

Plan czysty, liczniki się zgadzają? Odpal to samo bez --dry-run:

uv run --with asyncpg python migrate_data.py \
  --source-dsn "postgresql://befreeclub:DB_PASS@127.0.0.1:5434/bfc_admin_src" \
  --target-dsn "postgresql://befreeclub:DB_PASS@127.0.0.1:5434/befreeclub"

Kod wyjścia 0 i raport bez słowa ROZJAZD = dane skopiowane i zacommitowane (16 tabel, z zachowaniem id i sekwencji). Raport drukuje sam skrypt.

Co jeśli błąd
1 = rozjazd w weryfikacji (zrobiony rollback, baza nietknięta). 2 = błąd planu/SQL. Powtórka po naprawie: dodaj --truncate i potwierdź wpisując befreeclub. Pełny troubleshooting w scripts/data-migration/MIGRACJA_DANYCH.md.
# Spot-check po migracji
cd ~/repos/befreeclub/befreeclub/infra
docker compose exec db psql -U befreeclub -d befreeclub -c "SELECT count(*) FROM circle_dm.threads"
docker compose exec db psql -U befreeclub -d befreeclub -c "SELECT id, email FROM admin.users"
# Sprzątanie bazy tymczasowej
docker compose exec -T db dropdb -U befreeclub bfc_admin_src
rm ~/bfc_admin_*.dump

10Przepięcie DNS + Caddy + weryfikacja

Teraz kierujemy świat na nowy box. Najpierw dodajemy blok Caddy, potem przepinamy DNS, Caddy wyrabia certyfikat sam.

10a. Blok Caddy na nowym boxie

NOWY VPS

Dopisz do ~/caddy/Caddyfile:

admin.befreeclub.pro {
    @backend path /api/* /ws /health /health/*
    reverse_proxy @backend bfc-backend:3000
    reverse_proxy bfc-admin-front:80
}
cd ~/caddy
docker compose exec caddy caddy reload --config /etc/caddy/Caddyfile
Kolejność w bloku ma znaczenie
API, WebSocket i healthchecki łapie matcher @backend i idą do backendu. Cała reszta (SPA) do nginx. Matcher musi być PRZED catch-all - Caddy bierze pierwszy pasujący.

10b. Przełącz DNS

DNS - HOSTINGER

Zmień rekord A dla admin (host admin.befreeclub.pro) ze starego IP na NOWE_IP. TTL już masz obniżony do 300 s. Reszta rekordów (api, scoper, root) zostaje na starym boxie - ich nie ruszamy.

11Po przepięciu

Stary box zostaje jako rollback. Server admina jest już zatrzymany; bazę możesz też zatrzymać, ale nie kasuj wolumenów ani kontenerów, dopóki nowy panel nie pochodzi kilku dni w realnym użyciu.

STARY VPS
cd ~/repos/befreeclub/admin
docker compose stop admin-db      # opcjonalnie - baza też off, ale zostaje
Pamiętaj o rozjeździe danych
Wszystko, co zrobisz w NOWYM panelu po przepięciu (wysłane DM-y, statusy, notatki), nie istnieje w starej bazie. Rollback po godzinach pracy = utrata tych zmian. Dlatego rollback ma sens tylko tuż po cutover, nie po tygodniu.

!Bezpieczeństwo - co zabezpieczyć i gdzie

Część z tego zrobiłeś po drodze. To zbiorcza checklista dla boxa, który trzyma dane członków i (w fazie 2) klucze Stripe.

  • SSH tylko kluczem - bez hasła, bez roota (krok 3). To zamyka 99% automatycznych ataków.
  • Firewall ufw przepuszcza tylko 22/80/443. Sprawdź sudo ufw status.
  • fail2ban aktywny - banuje zgadywanie SSH.
  • Postgres nigdy publicznie - compose wystawia go tylko na 127.0.0.1:5434 (loopback). Z zewnątrz dostęp wyłącznie tunelem SSH. Nie zmieniaj tego na 0.0.0.0.
  • backend/.env z chmod 600, nigdy w gicie. To jedyne miejsce z sekretami.
  • Kontener backendu jako user uid=1000 (nie root) - już tak jest w obrazie. Wymóg Claude CLI i dobra higiena.
  • Snapshot OVH przed cutover - zrób snapshot nowego boxa tuż przed krokiem 9. Coś się sypie = przywracasz w minutę.
  • Automatyczne aktualizacje bezpieczeństwa (unattended-upgrades, krok 3).
  • Let's Encrypt auto-renew - Caddy odnawia cert sam, nic nie robisz.
  • Faza 2 - klucze Stripe live: backend ma bezpiecznik, w dev odmawia startu z kluczem sk_live_. Na prodzie NODE_ENV=production i live klucze są OK. Pilnuj, żeby dev i prod się nie pomyliły.
  • Dług do spłaty przy fazie 2: stary ADMIN_TOKEN był zahardkodowany w kodzie landinga (w gicie). Przy migracji landinga go zrotuj.

↻Backupy

Snapshot OVH (cały dysk) to jedno, ale chcesz też regularny dump samej bazy - mały, szybki do odtworzenia, łatwy do zrzucenia poza serwer.

NOWY VPS
# Skrypt: codzienny dump bazy, trzyma 14 ostatnich
mkdir -p ~/backups
cat > ~/backups/pg-backup.sh <<'EOF'
#!/bin/bash
cd ~/repos/befreeclub/befreeclub/infra
F=~/backups/befreeclub_$(date +%F).dump
docker compose exec -T db pg_dump -U befreeclub -d befreeclub -Fc > "$F"
ls -t ~/backups/befreeclub_*.dump | tail -n +15 | xargs -r rm
EOF
chmod +x ~/backups/pg-backup.sh
# Cron codziennie o 4:00
( crontab -l 2>/dev/null; echo "0 4 * * * ~/backups/pg-backup.sh" ) | crontab -
Zrzuć poza serwer
Backup, który leży na tym samym dysku co baza, nie chroni przed utratą serwera. Włącz w panelu OVH automatyczny backup VPS (Premium), albo dorzuć do skryptu wysyłkę dumpa na zewnętrzny storage (OVH Object Storage / inny).

STARY VPS
