Deploy na nowy VPS

Krok po kroku: stawiasz nowy serwer od zera, hartujesz go, wrzucasz panel admina (FastAPI + nginx + Postgres za Caddy) i przenosisz dane ze starego boxa. Landing robimy osobno w fazie 2. Komendy są do kopiowania. Checkboxy zapamiętują postęp w przeglądarce.

Legenda hostów
Przy każdym bloku komend jest etykieta gdzie to odpalić: NA MACU Twój laptop   NOWY VPS świeży serwer   STARY VPS obecny box z adminem   DNS panel Hostinger.
Podmieniaj WIELKIE_SŁOWA (np. NOWE_IP, DB_PASS) na swoje wartości.

iPrzegląd

W tej fazie przenosimy tylko panel admina (admin.befreeclub.pro). Stary VPS z adminem, befreeclub-api i gigscope zostaje włączony przez całą operację - to Twój rollback. Wygaszasz go dopiero, gdy nowy panel pochodzi kilka dni bez problemów.

Cel architektury na nowym boxie: trzy kontenery (bfc-db Postgres, bfc-backend API, bfc-admin-front nginx) w sieci bfc-net, a osobny Caddy hosta robi split routing i SSL na caddy_network.

iiZanim zaczniesz - co miej pod ręką

  • IP nowego VPS + hasło usera ubuntu (OVH daje gotowego usera ubuntu z sudo, nie roota). To NOWE_IP.
  • Dostęp SSH do starego VPS (do dumpa danych). To STARE_IP + Twój user.
  • Twój klucz SSH na Macu (~/.ssh/id_ed25519.pub). Jak nie masz: ssh-keygen -t ed25519.
  • Dostęp do DNS w Hostinger (domena befreeclub.pro, rekord A dla admin).
  • Circle Headless token (ten sam co w starym befreeclub-api / starym adminie, zmienna BOOTSTRAP_ADMIN_TOKEN). Bez niego Circle DM nie zaloguje się do Circle.
  • Login do Claude Max (zalogujesz CLI w kontenerze przez device flow).
  • Hasło DB starego boxa (DB_PASS w ~/repos/befreeclub/admin/.env na starym VPS) - do dumpa.
Zrób dzień wcześniej
Obniż TTL rekordu A admin.befreeclub.pro w Hostinger do 300 s (5 min). Wtedy przepięcie DNS w kroku 10 zadziała w minuty, nie w godziny.

2Klucz SSH na userze ubuntu

OVH dało Ci gotowego usera ubuntu z sudo, więc nie tworzymy nowego. Wgrywamy tylko do niego Twój klucz, żebyś z własnego Maca logował się bez hasła. Hasło do ubuntu zostaje aktywne jako awaryjny dostęp z innego komputera (krok 3).

NA MACU
# Skopiuj swój klucz publiczny na usera ubuntu (zapyta raz o hasło od OVH)
ssh-copy-id ubuntu@NOWE_IP
# Sprawdź, że z Twojego Maca wchodzi już bez hasła (kluczem)
ssh ubuntu@NOWE_IP
Wskazówka
Ze swojego Maca logujesz się jako ubuntu kluczem, bez hasła. Roota używasz przez sudo. Hasła ubuntu użyjesz tylko awaryjnie z obcego kompa.

Skrót: logowanie po nazwie zamiast IP

Żeby z Maca wpisywać ssh ubuntu@bfc-prod zamiast wklepywać IP, dodaj alias w swoim ~/.ssh/config. To tylko u Ciebie, nie wymaga żadnego DNS-u ani rekordu A.

NA MACU
# Podmień NOWE_IP na realne IP serwera przed wklejeniem
cat >> ~/.ssh/config <<'EOF'
Host bfc-prod
    HostName NOWE_IP
    User ubuntu
EOF
# Od teraz działa forma user@nazwa (dowolny user przed małpą):
ssh ubuntu@bfc-prod
Jak to działa
SSH dopasowuje blok po nazwie bfc-prod - część przed @ to user i nie wpływa na dopasowanie. Dlatego ssh ubuntu@bfc-prod i samo ssh bfc-prod robią to samo (linia User ubuntu to tylko domyślny user, gdy nie podasz nic przed @). Alias łapią też scp i rsync: scp plik.dump ubuntu@bfc-prod:~/. W dalszych krokach runbooka możesz ubuntu@NOWE_IP zastępować ubuntu@bfc-prod.

3Hardening SSH + firewall

Trzy rzeczy: blokujemy logowanie roota przez SSH (ale hasło dla ubuntu zostaje jako awaryjny dostęp z obcego kompa), zapora przepuszcza tylko SSH/HTTP/HTTPS, fail2ban banuje próby zgadywania hasła.

NOWY VPS
# Firewall - najpierw przepuść SSH, potem włącz (inaczej się zablokujesz)
sudo ufw allow OpenSSH
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw enable
sudo ufw status
# Hardening SSH: root nie loguje się przez SSH, hasło zostaje jako awaryjny dostęp dla ubuntu
sudo tee /etc/ssh/sshd_config.d/10-bfc.conf > /dev/null <<'EOF'
PermitRootLogin no
PasswordAuthentication yes
PubkeyAuthentication yes
EOF
sudo systemctl restart ssh
Czemu nazwa 10-bfc.conf
Obrazy OVH/cloud-init często wrzucają własny 50-cloud-init.conf, który potrafi ustawiać PasswordAuthentication. SSH bierze pierwszą napotkaną wartość, a pliki w sshd_config.d czyta alfabetycznie - 10- wygrywa z 50-, więc Twoje yes jest na wierzchu. Sprawdź po restarcie: sudo sshd -T | grep -i passwordauthentication ma pokazać yes.
Zanim zamkniesz terminal
Otwórz drugą sesję ssh ubuntu@NOWE_IP i sprawdź, że wchodzi. Jak działa - dopiero wtedy zamykaj pierwszą. Błąd w configu może wywalić restart sshd i wtedy nie wejdziesz żadną metodą. Awaryjnie zawsze masz konsolę KVM w panelu OVH. 
# fail2ban - domyślny jail SSH wystarcza, włącz i zostaw
sudo systemctl enable --now fail2ban
# Automatyczne aktualizacje bezpieczeństwa
sudo apt -y install unattended-upgrades
sudo dpkg-reconfigure -plow unattended-upgrades

5Caddy na hoście (reverse proxy + SSL)

Caddy stoi osobno i sam ogarnia certyfikaty Let's Encrypt. Na razie stawiamy go z pustą konfiguracją - blok domeny dodamy dopiero po wgraniu aplikacji (krok 10), bo cert wymaga działającego backendu i DNS celującego w nowy box.

NOWY VPS
mkdir -p ~/caddy && cd ~/caddy

Utwórz ~/caddy/docker-compose.yml:

name: caddy
services:
  caddy:
    image: caddy:2-alpine
    container_name: caddy
    restart: unless-stopped
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./Caddyfile:/etc/caddy/Caddyfile:ro
      - caddy-data:/data
      - caddy-config:/config
    networks:
      - caddy_network
volumes:
  caddy-data:
  caddy-config:
networks:
  caddy_network:
    external: true
# Pusty Caddyfile na start (blok admina dodasz w kroku 10)
echo "# bloki domen dodawane w kroku 10" > ~/caddy/Caddyfile
docker compose up -d
docker compose ps
Czemu pusto
Gdyby Caddy dostał teraz blok admin.befreeclub.pro, próbowałby wyrobić certyfikat, a DNS jeszcze wskazuje stary box. Najpierw aplikacja, potem DNS, na końcu blok Caddy.

6Kod na serwer

Monorepo trafia na serwer przez prywatne repo GitHub + deploy key (czysto i powtarzalnie przy aktualizacjach). Najpierw jednak musisz zrobić z katalogu repo i wypchnąć go na GitHub.

Najpierw .gitignore - krytyczne
Repo zawiera pliki .env z sekretami i .venv / node_modules. Nie mogą trafić do gita. Upewnij się, że root .gitignore wyklucza: .env, *.env (poza *.example), .venv/, node_modules/, data/, .dev-outbox/, __pycache__/.

6a. Commit i push na GitHub (Mac)

Załóż na GitHubie puste prywatne repo (np. befreeclub, bez README, żeby nie było konfliktu przy pierwszym pushu). Potem zrób z katalogu repo i wypchnij - commitujemy i pushujemy jako Ty.

NA MACU
cd ~/repos/befreeclub/migracja/befreeclub
git init
git add -A
git status            # SPRAWDŹ: żadnego .env na liście!
git commit -m "init befreeclub monorepo"
# Podłącz puste repo z GitHuba i wypchnij (idzie dostępem Twojego Maca do GitHuba)
git remote add origin git@github.com:TWOJ_LOGIN/befreeclub.git
git branch -M main
git push -u origin main

6b. Deploy key na serwerze

Serwer ma tylko czytać to jedno repo. Dlatego nie wrzucamy mu Twojego konta GitHub, tylko deploy key: parę kluczy SSH dającą dostęp do jednego repo, read-only. Jak kiedyś wycieknie, atakujący dostaje wgląd w to jedno repo, a nie w całe Twoje konto. Klucz prywatny zostaje na VPS, publiczny idzie na GitHub.

NOWY VPS
ssh-keygen -t ed25519 -f ~/.ssh/befreeclub_deploy -N "" -C "bfc-prod deploy key"
# -N "" = bez hasła do klucza, żeby git pull działał bez pytania (read-only na zamkniętym boxie = OK)
# Pokaż klucz PUBLICZNY do skopiowania (cała linia, z komentarzem na końcu):
cat ~/.ssh/befreeclub_deploy.pub
Bez wyświetlania, prosto do schowka Maca
Zamiast cat na VPS-ie możesz wyciągnąć klucz wprost do schowka Maca - odpal z Maca (nie w sesji SSH): ssh ubuntu@bfc-prod 'cat ~/.ssh/befreeclub_deploy.pub' | pbcopy. Schowek VPS-a i tak Ci się nie przyda, bo wklejasz w przeglądarce na Macu.

6c. Wklej klucz publiczny na GitHub

Na GitHubie: Twoje repo befreeclub → Settings (ustawienia repo, nie konta) → w lewym menu, sekcja Security, Deploy keys → Add deploy key:

  • Title: np. bfc-prod VPS (dowolna etykieta).
  • Key: wklej całą linię klucza publicznego (od ssh-ed25519 po komentarz na końcu).
  • Allow write access: zostaw odznaczone. Serwer tylko ciągnie kod, nigdy nie pushuje. Read-only.
  • Add key.

6d. Alias SSH, test, klon

Teraz mówisz SSH na VPS-ie, żeby do tego repo używał właśnie tego klucza. Alias github-bfc wskazuje na github.com, ale z konkretnym IdentityFile. IdentitiesOnly yes = próbuj tylko tym kluczem, nie podsuwaj innych (inaczej SSH potrafi przepalić limit prób, oferując po kolei wszystkie klucze).

NOWY VPS
cat >> ~/.ssh/config <<'EOF'
Host github-bfc
  HostName github.com
  User git
  IdentityFile ~/.ssh/befreeclub_deploy
  IdentitiesOnly yes
EOF
# Test PRZED klonem. Pierwszy raz potwierdź odcisk github.com wpisując 'yes'.
ssh -T git@github-bfc
Co ma się pokazać
Komunikat w stylu Hi TWOJ_LOGIN/befreeclub! You've successfully authenticated, but GitHub does not provide shell access. = sukces. Brak shell access jest normalny, GitHub tak zawsze odpowiada (nazwa repo w powitaniu potwierdza, że to deploy key przypięty do tego jednego repo). Odcisk ED25519, który akceptujesz (SHA256:+DiY3wvvV6TuJJhbpZisF/zLDA0zPMSvHdkr4UvCOqU), to prawdziwy klucz GitHuba - ten sam mechanizm sprawdzania tożsamości co przy SSL.
# Adres używa github-bfc (alias), nie github.com - to kieruje ruch przez deploy key.
mkdir -p ~/apps && cd ~/apps
git clone git@github-bfc:TWOJ_LOGIN/befreeclub.git befreeclub
cd befreeclub

Bo sklonowałeś aliasem, każdy późniejszy git pull z tego katalogu sam pójdzie tym samym kluczem. Nic więcej nie ustawiasz.

Bez GitHuba (alternatywa)
Jak nie chcesz repo na GitHub, wgraj kod bezpośrednio: rsync -av --exclude .venv --exclude node_modules --exclude '*.env' ~/repos/befreeclub/migracja/befreeclub/ ubuntu@NOWE_IP:~/apps/befreeclub/ z Maca. Aktualizacje robisz wtedy ponownym rsync zamiast git pull.

7Sekrety (.env)

Jeden plik sekretów: backend/.env. Compose czyta go i przez symlink infra/.env bierze DB_PASS do interpolacji.

NOWY VPS
cd ~/apps/befreeclub
cp backend/.env.production.example backend/.env
# Wygeneruj mocne hasło Postgresa
openssl rand -hex 24        # skopiuj wynik
# Master key do szyfrowania edytowalnych kluczy API w panelu (NOWY na prod, nie z dev)
openssl rand -base64 32 | tr '+/' '-_'     # 44-znakowy klucz Fernet - skopiuj wynik
nano backend/.env

W backend/.env ustaw na fazę 1 tylko to:

ZmiennaWartość
DB_PASSwynik openssl rand -hex 24
BOOTSTRAP_ADMIN_TOKENTwój Circle Headless token. Wymagany, żeby Circle DM mintował JWT i gadał z Circle.
OPENAI_API_KEYopcjonalnie - włącza transkrypcje głosówek i opisy zdjęć. Puste = te funkcje wyłączone, reszta działa.
SECRETS_MASTER_KEYwynik komendy openssl rand -base64 32 | tr ... powyżej (44 znaki). Szyfruje klucze API edytowalne w panelu (Połączenia API). Linia jest już w .env.production.example, tylko ją wypełnij. Puste = edycja kluczy w panelu wyłączona (cichy fallback na env, nie crash).

Wszystkie zmienne fazy 2 (STRIPE_*, RESEND_*, CIRCLE_*, SENDER_*, META_*) zostaw puste - landing migrujemy później, a backend startuje z nimi pustymi bez błędu.

# Symlink, żeby compose widział DB_PASS przy interpolacji
cd infra
ln -s ../backend/.env .env
# Zamknij plik sekretów przed światem
chmod 600 ../backend/.env
Reguła sekretów
backend/.env nigdy nie wchodzi do gita ani nigdzie poza ten serwer. chmod 600 = czyta go tylko Twój user. Sprawdź: ls -l backend/.env ma pokazać -rw-------.

9Migracja danych ze starego boxa

To moment z oknem serwisowym. Zatrzymujemy starego admina (żeby dane były ostateczne), robimy dump, przenosimy na nowy box, wgrywamy do bazy tymczasowej i odpalamy skrypt mapujący do nowych schematów.

Czego skrypt nie rusza
Tabela admin.encrypted_secrets (klucze API z panelu) to nowa, pusta tabela - nie jest migrowana. Stary admin jej nie miał. Cztery klucze (OpenAI/Resend/Sender/Meta) wpiszesz w panelu dopiero w fazie 2, albo lecą na fallbacku z .env.

STARY VPS

9b. Przenieś dump na nowy box

NA MACU

9c. Wgraj dump do bazy tymczasowej na nowym boxie

NOWY VPS

9d. Uruchom skrypt migracji

Skrypt mapuje stare tabele (public.*) na nowe schematy (admin.*, circle_dm.*) i przenosi je do bazy befreeclub. Odpalamy go na hoście, oba Postgresy widzi po loopbacku 127.0.0.1:5434.

NOWY VPS
# uv do odpalenia skryptu (jednorazowo)
curl -LsSf https://astral.sh/uv/install.sh | sh
source ~/.bashrc
cd ~/apps/befreeclub/scripts/data-migration
# DRY-RUN: pokazuje plan i liczniki, nic nie zapisuje. DB_PASS = to z backend/.env
uv run --with asyncpg python migrate_data.py \
  --source-dsn "postgresql://befreeclub:DB_PASS@127.0.0.1:5434/bfc_admin_src" \
  --target-dsn "postgresql://befreeclub:DB_PASS@127.0.0.1:5434/befreeclub" \
  --dry-run

Plan czysty, liczniki się zgadzają? Odpal to samo bez --dry-run:

uv run --with asyncpg python migrate_data.py \
  --source-dsn "postgresql://befreeclub:DB_PASS@127.0.0.1:5434/bfc_admin_src" \
  --target-dsn "postgresql://befreeclub:DB_PASS@127.0.0.1:5434/befreeclub"

Kod wyjścia 0 i raport bez słowa ROZJAZD = dane skopiowane i zacommitowane (16 tabel, z zachowaniem id i sekwencji). Raport drukuje sam skrypt.

Co jeśli błąd
1 = rozjazd w weryfikacji (zrobiony rollback, baza nietknięta). 2 = błąd planu/SQL. Powtórka po naprawie: dodaj --truncate i potwierdź wpisując befreeclub. Pełny troubleshooting w scripts/data-migration/MIGRACJA_DANYCH.md.
# Spot-check po migracji
cd ~/apps/befreeclub/infra
docker compose exec db psql -U befreeclub -d befreeclub -c "SELECT count(*) FROM circle_dm.threads"
docker compose exec db psql -U befreeclub -d befreeclub -c "SELECT id, email FROM admin.users"
# Sprzątanie bazy tymczasowej
docker compose exec -T db dropdb -U befreeclub bfc_admin_src
rm ~/bfc_admin_*.dump

10Przepięcie DNS + Caddy + weryfikacja

!Bezpieczeństwo - co zabezpieczyć i gdzie

Część z tego zrobiłeś po drodze. To zbiorcza checklista dla boxa, który trzyma dane członków i (w fazie 2) klucze Stripe.

  • SSH bez roota - root nie loguje się przez SSH (krok 3). Z Twojego Maca wchodzisz kluczem; hasło ubuntu zostaje jako awaryjny dostęp z obcego kompa. Mocne, generowane hasło + fail2ban pilnują portu.
  • Firewall ufw przepuszcza tylko 22/80/443. Sprawdź sudo ufw status.
  • fail2ban aktywny - banuje zgadywanie SSH.
  • Postgres nigdy publicznie - compose wystawia go tylko na 127.0.0.1:5434 (loopback). Z zewnątrz dostęp wyłącznie tunelem SSH. Nie zmieniaj tego na 0.0.0.0.
  • backend/.env z chmod 600, nigdy w gicie. To jedyne miejsce z sekretami.
  • Kontener backendu jako user uid=1000 (nie root) - już tak jest w obrazie. Wymóg Claude CLI i dobra higiena.
  • Snapshot OVH przed cutover - zrób snapshot nowego boxa tuż przed krokiem 9. Coś się sypie = przywracasz w minutę.
  • Automatyczne aktualizacje bezpieczeństwa (unattended-upgrades, krok 3).
  • Let's Encrypt auto-renew - Caddy odnawia cert sam, nic nie robisz.
  • Faza 2 - klucze Stripe live: backend ma bezpiecznik, w dev odmawia startu z kluczem sk_live_. Na prodzie NODE_ENV=production i live klucze są OK. Pilnuj, żeby dev i prod się nie pomyliły.
  • Dług do spłaty przy fazie 2: stary ADMIN_TOKEN był zahardkodowany w kodzie landinga (w gicie). Przy migracji landinga go zrotuj.

?Dodatek: jak działa SSL i Let's Encrypt

To nie jest potrzebne do samego deployu - Caddy robi wszystko sam. Ale warto rozumieć, co się dzieje pod spodem, choćby po to, żeby kiedyś komuś wytłumaczyć. Całość na przykładach z życia.

Po co SSL - problem do rozwiązania

Twoje dane lecą przez internet jak pocztówka: od przeglądarki do serwera przechodzą przez kilkanaście rąk (router, dostawca, routery szkieletowe, hostingownia). Każdy „listonosz" po drodze może ją przeczytać (hasło, numer karty), a nawet dopisać coś od siebie. SSL (dziś technicznie TLS; HTTPS = HTTP zapakowany w TLS) zamienia tę pocztówkę w zaklejoną, plombowaną kopertę. Daje trzy rzeczy: szyfrowanie (listonosze widzą bełkot), tożsamość (gadasz z prawdziwym serwerem, nie podszywaczem) i nienaruszalność (zmiana po drodze zostanie wykryta).

Problem 1: jak ustalić wspólny klucz, skoro wszyscy podsłuchują

Żeby szyfrować, obie strony potrzebują wspólnego sekretu. Ale jak się na niego umówić przez kabel, który każdy podsłuchuje? Rozwiązanie: kłódki. Serwer rozdaje na lewo i prawo otwarte kłódki (klucz publiczny). Każdy może zatrzasnąć kłódkę na pudełku (zaszyfrować), ale otworzyć potrafi tylko serwer, bo tylko on ma kluczyk (klucz prywatny, nigdy nie opuszcza serwera). Nawet jak podsłuchiwacz widzi, jak zamykasz pudełko, to mu to nic nie daje - patrzył na zamykanie, otworzyć nie umie.

Problem 2: skąd wiesz, że ta kłódka jest naprawdę od tego serwera

Atakujący może podstawić swoją kłódkę i powiedzieć „to ja, befreeclub.pl". Zamkniesz hasło w jego pudełku - klasyczny man in the middle. Rozwiązanie: paszport. Każdy może wydrukować kartkę „jestem Tomasz", ale paszport jest podstemplowany przez państwo, któremu wszyscy ufają. Celnik nie zna Cię osobiście - ufa pieczęci.

  • Paszport = certyfikat: klucz publiczny serwera + nazwa domeny + podpis zaufanej instytucji.
  • Państwo = CA (Certificate Authority), urząd certyfikacji.
  • Twoja przeglądarka ma wbitą na sztywno listę zaufanych CA (lista „rządów, których paszporty akceptuję").

Przeglądarka sprawdza: podpisany przez CA, któremu ufam? Na tę domenę? Jeszcze ważny? Jak gra - kłódeczka w pasku. Podpisu CA nie da się podrobić, bo opiera się na tej samej matematyce kłódek (CA podpisuje swoim kluczem prywatnym, każdy weryfikuje publicznym, wbitym w przeglądarkę).

Mit do obalenia
Kłódeczka nie znaczy „strona uczciwa", tylko „połączenie jest prywatne i naprawdę gadasz z TĄ domeną". Strona phishingowa też może mieć ważny certyfikat. Paszport potwierdza tożsamość, nie charakter.

Skąd CA wie, że certyfikat należy się właśnie Tobie

Bo inaczej każdy wyrobiłby sobie cert na google.com. Zanim CA podpisze, każe udowodnić, że kontrolujesz domenę (domain validation). Realny przykład: chcesz pozwolenie na dom, urzędnik mówi „przyklej tę dokładną karteczkę na drzwiach, podjadę sprawdzić". Dwa sposoby:

  • HTTP-01: „połóż plik o tej treści pod http://.../.well-known/acme-challenge/XYZ". Umiesz = kontrolujesz serwer domeny.
  • DNS-01: „dodaj ten rekord TXT do DNS". Dowód, że kontrolujesz DNS domeny.

Let's Encrypt i ACME - czemu to się zautomatyzowało

Kiedyś cert to była orka: generujesz klucz, wypełniasz formularz, płacisz, wgrywasz pliki, i pamiętasz, żeby co roku odnowić. Zapomniałeś = wygasły cert = czerwony ekran „połączenie nie jest prywatne". Let's Encrypt (od 2015) to darmowy, zautomatyzowany CA. ACME to protokół, w którym serwer sam dogaduje się z Let's Encrypt: „potrzebuję certu" → „udowodnij, połóż ten plik" → kładzie → „masz cert". Bez człowieka, bez płacenia. Certy żyją krótko (90 dni), bo odnawianie jest automatyczne - krótkie życie = mniejsza szkoda przy wycieku klucza.

Jak ogarnia to Caddy (i czemu kolejność w tym runbooku jest taka)

Caddy ma ACME wbudowane. Samym wpisaniem domeny w Caddyfile mówisz „obsłuż po HTTPS", a on dalej sam: widzi, że potrzebuje certu → gada z Let's Encrypt → odpowiada na wyzwanie HTTP-01 (nasłuchuje na 80/443, więc sam podstawia magiczny plik) → dostaje cert → zapisuje w wolumenie caddy-data → serwuje HTTPS → odnawia sam ~30 dni przed końcem. Nigdy tego nie dotykasz.

Tu się spina cały runbook
Wyzwanie ACME musi do Caddy dolecieć - dlatego firewall przepuszcza 80/443 (krok 3), a blok domeny dodajesz dopiero po przepięciu DNS (krok 10). Gdyby Caddy dostał blok zanim DNS celuje w nowy box, wyzwanie poleciałoby na stary serwer i cert by nie powstał. Stąd sztywna kolejność: aplikacja stoi → DNS przepięty → blok Caddy.

Czemu na WordPressie na hostingu czasem trzeba SSL ustawiać ręcznie

Bo SSL nie ma nic wspólnego z samym WordPressem. Cert siedzi na warstwie serwera WWW (Apache/nginx) albo na proxy przed nim, nie w PHP. Pytanie brzmi: czy hosting zautomatyzował ten taniec ACME.

  • Stare / tanie hostingi: nie automatyzowały. Kupowałeś cert, wchodziłeś w cPanel, ręcznie wklejałeś cert i klucz, ustawiałeś przekierowania. Plus WordPressowi trzeba było powiedzieć „używaj https" (adres strony w ustawieniach) i często doinstalować wtyczkę na mixed content (strona na https, ale obrazki linkowane po http, co przeglądarka blokuje).
  • Dziś / normalne hostingi: większość (cPanel AutoSSL, Plesk) ma wbudowane Let's Encrypt i robi to samo co Caddy - automatycznie. Dlatego coraz częściej już nie trzeba ręcznie.

Czyli to nigdy nie był „problem WordPressa", tylko „czy warstwa serwera ma automat ACME". Caddy ma go z pudełka. Nowoczesny hosting pod spodem odpala dokładnie ten sam Let's Encrypt, tylko Ci tego nie pokazuje.

Logika w jednym zdaniu
Kłódki rozwiązują „jak ustalić sekret jawnie", paszport + urząd (certyfikat + CA) rozwiązują „komu ufać", karteczka na drzwiach (challenge) to dowód, że domena Twoja, a ACME / Let's Encrypt / Caddy to robot, który ten cały urzędowy taniec odwala za Ciebie co 60 dni.
